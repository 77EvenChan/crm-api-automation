## 📌项目背景与定位
本项目是我在不断精进测试技能时，从0到1独立设计的**企业级接口自动化测试底座**，针对于传统接口测试中存在的“环境依赖强、配置管理混乱、执行效率低、用例维护成本高”等痛点，我引入了现代Python最佳实践与API Object防腐层架构，实现了真正的**测试左移**
。

此仓库展示的是框架的**核心引擎**与**基准测试链路**，就是说提供了一个开箱即用、易于接入CI/CD的自动化解决方案。

## 🚀 核心架构与技术栈
* **语言标准**：Python 3.10+
* **核心驱动**：Pytest 9.0.0+
* **网络引擎**：Requests（深度封装HTTPAdapter与Session复用）
* **配置中心**：Pydantic & Pydantic-Settings（抛弃传统YAML，使用强类型与云原生）
* **左移拦截**：Requests-Mock（底层Socket级业务数据注入）
* **报告呈现**：Allure Framework

## 架构设计与本项目的心得
我一直在尝试精进我的测试技能或者说想在测试或者IT这行再走远一些，于是踩了不少坑，也慢慢摸索出了一套自己觉得很有意义的心得。

**第一：在项目结构和配置管理上，我彻底转向了pyproject.toml+src布局**：
以前用setup.py和requirements.txt，但我换环境git时发现，总是出现缺包的情况，而且配置分散，改起来很麻烦，所以我接触了src/crm_api这种布局，避免了路径污染，也能让其他人一眼就看明白这是CRM项目的核心模块，以后真要发到PyPI也没问题，配置管理我也换成了pydantic-settings，不用再自己手写YAML解析了，还能直接从环境变量读取配置，而且类型校验帮我提前拦住了不少配置错误。

**第二：开发思路上我坚持“基础设施先行”和“测试左移”**：
在过去我学测试框架工程化思维时，就有不少B站老师总是先教你如何找公共API或者如何买他们的SUT，但实际上是推销自己的课程，我也尝试过很多次等后端完成再进行搭建测试，对此我对于那些乱授课的B站老师们真的是不屑一顾，总是把人带到沟里去，经过我反复琢磨，最终发现一种思路：**基础设施先行和测试左移**，在未来工作中我也会坚持：先把底座搭好，像HTTP客户端、异常封装、日志拦截这些先做稳定，这样后端一给接口，我就能立刻开始写业务封装，两边并行，或者测试先行，效率会高很多。比如HTTP客户端，我直接用Session（）复用连接池，配合Keep-Alive，执行效率提升非常明显，再也不用担心网络抖动导致的误报，总之坚持基础设施先行和测试左移是每一个测试工程师都应了解的开发思路，

**第三：分层设计上我引入了Service层，把业务和细节隔离开**：
我来解释一下，就是说用例层只关心业务逻辑，完全不知道底层是HTTP还是其他的什么，比如后端突然改了接口路径，那我只需要改Service层那里的一处，所有用例自动适配，这维护成本瞬间就降低了，断言我也做了升级，不光看HTTP状态码，而是拿业务码和数据库状态一起校验，甚至还会提取customer_id来确认数据是否真的落盘了，这样覆盖也能更加全面，对项目对公司也是一种负责的态度。

**第四：代码的可读性和复用性的提升**：
我把接口文档直接写在类注释里，这样不管是新人还是同事啊经理啊接手或者查阅代码时不用翻YApi了，看代码就能明白业务流转，数据驱动用pytest的parametrize，同样的测试逻辑跑不同的数据，代码复用率很高，也减少了重复劳动。

💻 快速启动指南
1. 环境准备
建议使用python3.10及以上版本并创建虚拟环境：
```Bash
python -m venv venv
venv/Scripts/activate
```
2. 极速安装依赖
本项目采用现代化的pyproject.toml，摒弃了过时的requirements.txt
```Bash
pip install -e .
(推荐上面的命令)，或者
pip install .
```
3. 运行测试并生成报告
```Bash
# 执行业务用例，并收集Allure原始数据
pytest tests/ -v -s --alluredir=./allure-results

# (可选) 如果本地已安装Allure命令行工具，可以直接启动报告服务
allure serve ./allure-results
```Bash
